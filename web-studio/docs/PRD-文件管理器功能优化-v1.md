# PRD：Viking File Manager 功能优化（v1）

- 文档版本：v1.0
- 日期：2026-04-09
- 负责人：待定
- 适用范围：`web-studio` 文件管理器（三栏：目录树 / 文件列表 / 预览）

---

## 1. 背景与问题

当前文件管理器已具备基础浏览与预览能力，但在以下方面仍影响效率：

1. 目录内容较多时，用户缺少快速筛选能力。
2. 高频访问目录无法一键抵达，重复展开目录树成本高。
3. 管理能力不足（新建/重命名/删除目录），从“可浏览”到“可管理”尚未形成闭环。
4. 目录变更后，列表、目录树、预览之间需要更明确的一致性保障。

> 备注：顶部面包屑已存在，本期不重复建设。

---

## 2. 目标（Goals）

### 2.1 产品目标

- 提升“找文件/找目录”的效率。
- 提升高频目录访问效率。
- 建立最小可用的目录管理闭环。
- 保证目录操作后的数据一致性与可预期反馈。

### 2.2 量化目标（上线后 2~4 周观察）

- 平均定位目标文件时间下降 30%。
- 高频目录访问点击步骤下降 40%。
- 目录管理操作成功率 ≥ 98%。
- 目录操作后数据不一致反馈（用户手动刷新/报错）下降 50%。

---

## 3. 非目标（Non-Goals）

本期不包含：

- 批量操作（批量删除/移动/下载）
- 文件内容编辑器能力增强（高级 diff、协作编辑等）
- 权限系统重构
- 跨存储后端抽象改造

---

## 4. 用户故事（User Stories）

1. 作为用户，我希望输入关键词快速筛选当前目录内容，减少滚动查找。
2. 作为用户，我希望收藏常用目录，下次可直接进入。
3. 作为用户，我希望在界面内直接新建、重命名、删除目录，不需要切换工具。
4. 作为用户，我希望每次目录操作后，列表/目录树/路径状态保持一致且有明确反馈。

---

## 5. 需求范围（Scope）

## 5.1 功能 A：目录内筛选（必做）

### 功能说明

在“文件列表”区域增加筛选能力：

- 名称关键词筛选（实时）
- 类型筛选：全部 / 仅目录 / 仅文件
- 文件类型筛选（可选）：Markdown / 图片 / 代码 / 其他

### 交互要求

- 筛选仅作用于当前目录，不跨目录。
- 空结果展示“无匹配结果”，并提供“一键清空筛选”。
- 刷新目录后保留筛选条件（直到用户主动清空）。

### 验收标准

- 对 500 项以内目录，输入筛选关键字后 200ms 内完成渲染更新。
- 多条件组合筛选结果正确。

---

## 5.2 功能 B：收藏目录（必做）

### 功能说明

- 用户可将当前目录加入收藏/取消收藏。
- 在左侧树上方提供“收藏目录”快捷区。
- 支持点击收藏项快速跳转。

### 数据规则

- 收藏基于目录 URI 唯一。
- 目录被删除或不可达时，收藏项展示“失效”状态，并可移除。
- 收藏信息按用户维度保存（本地持久化为最低要求，服务端存储为可扩展项）。

### 验收标准

- 收藏/取消收藏即时生效。
- 重新进入页面后收藏列表可恢复。

---

## 5.3 功能 C：目录管理能力（必做）

### C1 新建目录
- 入口：当前目录工具栏 + 右键菜单（至少一个入口）。
- 输入：目录名（必填，校验非法字符、重复名）。
- 成功：提示成功并自动刷新当前目录和目录树。

### C2 重命名目录
- 入口：目录项操作菜单。
- 输入：新名称（必填、合法、不能与同级重名）。
- 成功：更新当前选中路径与面包屑（若影响当前路径）。

### C3 删除目录
- 入口：目录项操作菜单。
- 交互：二次确认（明确提示是否递归删除）。
- 规则：
  - 非空目录默认要求“递归删除”确认。
  - 删除当前目录时，自动跳转到父目录。

### 验收标准

- 新建/重命名/删除均有明确 loading、成功、失败反馈。
- 失败时展示可理解原因（权限不足、路径不存在、冲突等）。
- 操作完成后列表、目录树、预览状态一致。

## 6. 后端 API 现状与对接

根据当前 OpenAPI 生成代码，可用接口如下：

- 列表：`GET /api/v1/fs/ls`
- 目录树：`GET /api/v1/fs/tree`
- 目录信息：`GET /api/v1/fs/stat`
- 新建目录：`POST /api/v1/fs/mkdir`（`MkdirRequest { uri }`）
- 删除：`DELETE /api/v1/fs?uri=...&recursive=...`
- 移动/重命名：`POST /api/v1/fs/mv`（`MvRequest { from_uri, to_uri }`）
- 内容读取：`GET /api/v1/content/read`
- 内容下载：`GET /api/v1/content/download`
- 内容写入（文件）：`POST /api/v1/content/write`

## 待后端确认（必须）

1. `mkdir` 对“已存在目录”返回码与错误语义。
2. `mv` 在跨目录/同目录重命名时的一致语义。
3. `delete` 删除非空目录时，`recursive=false` 的错误码与错误体。
4. 并发写操作冲突策略（后写覆盖/拒绝/版本控制）。
5. 统一错误码规范（409 冲突、403 权限、404 不存在等）。

---

## 7. 数据一致性策略（关键）

目录变更类操作（mkdir/mv/delete）统一遵循：

1. **先后端成功，后前端更新**（保守策略，优先正确性）
2. 成功后统一触发：
   - 当前目录列表刷新
   - 目录树刷新
   - 涉及路径状态修正（当前目录、选中文件、预览）
3. 若当前预览文件被删除/移动：
   - 清空预览并提示“文件已变更，请重新选择”
4. 若操作失败：
   - 不更新本地状态
   - 保留用户输入并展示错误原因 + 重试入口

---

## 8. 交互与文案要求

- 所有危险操作（删除）必须二次确认。
- 文案统一：
  - 成功：`已创建目录` / `已重命名` / `已删除`
  - 失败：`操作失败：{原因}`
- 不可逆操作必须出现风险提示（尤其递归删除）。

## 11. 风险与应对

1. **后端语义不明确导致前端分支复杂**
   - 应对：先对齐错误码契约文档，再联调。
2. **删除/移动后 UI 残留旧状态**
   - 应对：统一 mutation 成功后的刷新与状态修正流程。
3. **大目录筛选性能抖动**
   - 应对：限制首期范围（当前目录内、500 项以内），并做性能监控。

---

## 12. 发布准入（Go/No-Go）

满足以下条件才可发布：

- 三个核心功能（筛选、收藏、目录管理）验收通过。
- 目录操作一致性场景（当前目录被删、重命名后路径切换）回归通过。
- 关键埋点可采集。
- 已完成后端错误码与语义对齐。
