# Security Assessment Finding

## Finding ID
OVK-A01-2026-001

## Title
Unauthenticated Access to Bot Proxy Endpoints (`/bot/v1/chat`, `/bot/v1/chat/stream`)

## Severity
**High**

## OWASP Classification
**OWASP Top 10 (2021): A01 – Broken Access Control**



## CWE Mapping
- **Primary CWE:** CWE-306 — Missing Authentication for Critical Function
- **Related CWE:** CWE-862 — Missing Authorization

## Executive Summary
During review of the bot proxy implementation, we identified that requests to `/bot/v1/chat` and `/bot/v1/chat/stream` can be processed without mandatory authentication checks when bot proxying is enabled (`with_bot=true`).

The current implementation optionally forwards credentials if provided, but does not enforce authentication at the OpenViking API boundary before proxying to the backend bot service.

## Technical Description
The bot router is mounted at `/bot/v1`, exposing proxy handlers for chat and streaming chat operations. The handlers call `verify_auth()` to read `X-API-Key` or `Authorization` headers; however, this function is non-enforcing and returns `None` when credentials are absent. The request flow continues and is forwarded upstream regardless.

As a result, unauthenticated callers may invoke backend bot functionality through the OpenViking proxy layer.

## Affected Endpoints
- `POST /bot/v1/chat`
- `POST /bot/v1/chat/stream`

## Conditions for Exploitation
Exploitation is most likely when all of the following are true:
1. `with_bot=true`
2. The service is reachable from untrusted clients (internet or broad internal network)
3. No external compensating access control (API gateway auth, mTLS gateway, strict network ACL)

## Impact Assessment
Potential impacts include:
- Unauthorized consumption of LLM/tooling services
- Cost amplification through abusive token/compute usage
- Exposure of sensitive output generated by backend bot workflows

Given the low complexity of exploitation (single HTTP request) and direct route reachability when enabled, severity is assessed as **High**.

## Reproduction (PoC)
```bash
# Unauthenticated request to chat endpoint
curl -i -X POST http://<host>:1933/bot/v1/chat \
  -H 'Content-Type: application/json' \
  -d '{"message":"hello"}'
```

```bash
# Unauthenticated request to streaming endpoint
curl -N -X POST http://<host>:1933/bot/v1/chat/stream \
  -H 'Content-Type: application/json' \
  -d '{"message":"hello"}'
```

Observed vulnerable behavior: request is forwarded instead of being rejected with `401/403` at the proxy boundary.

## Recommended Remediation
1. Enforce authentication dependency on bot routes (e.g., `Depends(get_request_context)`).
2. Apply explicit authorization policy (e.g., role checks for `USER`/`ROOT`) as appropriate to business logic.
3. Enforce fail-closed behavior: return `401/403` for missing/invalid credentials before upstream forwarding.
4. Add automated regression tests:
   - unauthenticated `/bot/v1/chat` returns `401/403`
   - unauthenticated `/bot/v1/chat/stream` returns `401/403`
   - authenticated requests are proxied successfully

## Interim Mitigations
Until code remediation is deployed:
- Enforce authentication at edge/gateway layer
- Restrict network exposure of `/bot/v1/*`
- Apply rate limiting and anomaly monitoring for bot proxy routes

## Confidence
**High** – The issue is directly evidenced by route and handler control flow.
