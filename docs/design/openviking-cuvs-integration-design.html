Integration Architecture · v0.1

OpenViking × NVIDIA cuVS
集成架构设计

在不重写 OpenViking 本地存储、标量索引、稀疏检索和恢复机制的前提下， 以可插拔方式将 dense vector search 切换到 NVIDIA cuVS。首阶段优先验证功能正确性， 后续再围绕索引更新、过滤、并发与 GPU 资源利用做性能化演进。

功能验证实现已完成，待真实 GPU 验证 日期：2026-07-02 OpenViking 基线：ea09846 默认算法：brute_force

01. 结论摘要

cuVS 是 GPU 向量索引库，不是带 CRUD、标量查询和持久化能力的完整数据库。 因此本设计将它放在 OpenViking 本地索引内部，只替换 dense 检索执行器； OpenViking 继续负责上层 API、权威记录、生命周期和兼容语义。

A

集成边界

切在 LocalIndex.search() 的 dense 分支，避免复制整个 Collection 后端。

B

权威状态

本地 Store 仍是唯一事实来源；cuVS GPU 索引是可随时重建的派生状态。

C

正确性优先

首版采用变更后标脏、下一次查询批量重建，先保证 upsert/delete 语义。

D

过滤策略

支持的过滤先在 CPU 计算 bitset，再作为 cuVS prefilter；不采用不可靠的 post-filter。

E

兼容回退

稀疏/混合查询及暂不支持的过滤可回退原生索引，聚合和标量排序始终保留原生路径。

F

失败原则

缺包、无 GPU 或 cuVS 运行错误直接暴露；只有“能力不支持”才进行语义回退。

一句话架构： OpenViking 管数据与语义，cuVS 管 GPU dense top-k；两者通过 label、host shadow 和 filter bitset 对接。

02. 目标与边界

本阶段目标

  • 通过 backend: "cuvs" 显式启用，不改变上层检索 API。
  • dense query 真实进入 cuVS brute-force 或 CAGRA。
  • 兼容 cosine、inner product、L2 的现有分数定义。
  • 支持常用租户、路径、数值和列表过滤。
  • upsert、delete、进程重启后仍返回正确结果。
  • 保留 native 路径作为能力补集与正确性参照。

本阶段不做

  • 不声称已获得性能收益或生产级性能数据。
  • 不以 cuVS 序列化文件作为权威持久化格式。
  • 不移除 OpenViking 原生 dense index。
  • 不实现多 GPU、跨机分片或 GPU 资源池。
  • 不在第一版解决写密集场景的增量维护。
  • 不重写 sparse/hybrid、aggregate 和复杂标量执行器。

03. 总体架构

新增的 cuVS 层与原生索引并存。路由发生在本地索引层，因而 Collection 的记录管理、 字段补全、offset/limit、结果回表和持久化协议不需要分叉。

图 1 · 当前功能验证版架构。绿色路径是新增 GPU dense 路径，蓝色路径是保留的 OpenViking 原生能力。

组件职责

组件职责状态

数据所有权与派生状态

① Local Store

权威、可持久化。保存记录、向量和字段；恢复从这里开始。

② Native Index

派生状态。由 snapshot + delta 管理，承担完整本地能力。

③ Host Shadow

派生状态。按 label 保存 dense vector 与过滤字段，用于重建和 bitset。

④ GPU Index

易失派生状态。首次查询或 dirty 后按需重建，不作为持久化格式。

OpenViking 的字符串主键先按既有规则映射为 uint64 label。cuVS 返回的是 GPU 数据集行号， CuVSDenseIndex 再通过构建时的 label 数组映射回 OpenViking label，最后由 Collection 回表得到完整记录。

04. 核心链路

4.1 写入与删除

1. API / Adapter接收 upsert/delete，完成基础请求转换。
2. Collection校验 schema、生成 label，并更新权威 Store/delta。
3. Native Index按既有逻辑 upsert/delete，维持 fallback 可用。
4. Host Shadow更新或删除 label 对应的数据，并设置 dirty=true。
5. Next Query批量重建 GPU index 后查询，保证完成写入后的可见性。
当前策略是功能版折中： 每次写入不立即修改 CAGRA，而是标脏后在下一次 dense 查询整体重建。这样逻辑简单且删除语义明确， 但写频繁时首查延迟和 GPU 构建成本会很高。

4.2 查询路由

图 2 · 请求级能力路由。GPU 运行异常不走这条 fallback 分支，而是直接向上暴露。

cuVS 路径内部按以下顺序执行：

  1. 在 host shadow 上求值过滤 DSL，得到与当前数据行顺序一致的 eligible mask。
  2. 将 bool mask 打包为 cuVS 需要的 uint32 bitset。
  3. 若 dirty，则复制 dense matrix 到 GPU 并重建 brute-force/CAGRA index。
  4. 执行 top-k；CAGRA 的 itopk_size 至少提升到不小于 k、并按 32 向上取整的值。
  5. 将 cuVS row id 映射为 label，转换距离分数，再交给 Collection 做回表和 offset 截取。

4.3 进程重启与持久化恢复

1. 读取 metadata恢复 Collection 与 index 定义。
2. 恢复 native加载 snapshot，并按已有协议 replay delta。
3. 扫描 Store读取该 index 的当前 CandidateData 全量。
4. 填充 shadow恢复 label、vector 和 filter fields，保持 GPU index 未构建。
5. 首次查询按配置构建 GPU index 并开始检索。

不保存 cuVS index 文件是刻意的设计选择：它避免将 OpenViking 的数据可恢复性绑定到 GPU 型号、CUDA/cuVS 版本和索引序列化兼容性。 代价是重启后的第一次 dense 查询需要承担构建成本；后续可引入受版本约束的 cache，但不能把 cache 升格为权威数据。

05. 兼容语义

5.1 距离与返回分数

OpenViking 配置cuVS 执行对外 score
cosine数据与 query 做 L2 normalize，然后执行 inner_productinner product，越大越相似
inner_productinner_productinner product，越大越相似
l2sqeuclidean1 - squared_l2，保持现有本地后端定义

5.2 Dtype 与非侵入性边界

保留 CPU 默认行为： 启用 cuVS 不修改默认 backend，也不迁移或重写 native index。native CPU 继续使用 VectorIndex.Quant=int8（逐向量 scale），cuVS GPU shadow 当前使用 float32。因此两条 dense 路径不是等 dtype、等内存或完全相同数值语义的比较； native fallback 始终走原有 int8 index，benchmark 需要显式报告 dtype 与 Recall@K。

5.3 过滤与能力矩阵

能力执行路径说明
and / orcuVS prefilter在 CPU 递归组合 mask。
must / must_notcuVS prefilter支持标量值与列表值；覆盖常见 account_id 租户过滤。
containscuVS prefilter在字段值上计算包含关系。
range / range_outcuVS prefilter支持可比较的数值/标量范围。
path prefix / depthcuVS prefilter保持 OpenViking 路径层级过滤。
date_time / geo_pointnative fallback首版不在 Python evaluator 重复复杂字段转换语义。
sparse / hybridnativecuVS 只接管 pure dense；混合查询保留现有引擎。
scalar sort / aggregatenative不属于 cuVS dense top-k 的职责。
为什么不做 post-filter： 先取未过滤 top-k 再过滤会在高选择性条件下少返回、错过真实 top-k；无限 over-fetch 也无法给出稳定正确性。 因此支持的条件必须成为 cuVS 搜索前的 bitset，不能安全表示的条件直接交回 native。

依赖选择

cuVS 26.06 Python wheel 要求 Python 3.11+。CUDA 12 与 CUDA 13 使用互斥的 wheel， 因此不把二者同时声明为项目 extra，部署时按机器 CUDA 大版本显式安装： 

# CUDA 12
pip install -e .
pip install cuvs-cu12 'cupy-cuda12x[ctk]' --extra-index-url=https://pypi.nvidia.com

# CUDA 13
pip install -e .
pip install cuvs-cu13 'cupy-cuda13x[ctk]' --extra-index-url=https://pypi.nvidia.com

[ctk] 会补齐 cuVS Python 互操作路径需要的 CUDA toolkit headers；只有 driver、没有完整 toolkit 的宿主环境也可直接运行。

07. 一致性、并发与故障策略

READ-AFTER-WRITE

upsert/delete 完成后 host shadow 已标脏，下一次 dense 查询会先重建，因此顺序调用具备读己之写语义。

并发可见性

CuVSDenseIndex 用单个重入锁保护 shadow、重建与 search；同一 index 的 GPU 查询当前被串行化。与 native/Store 之间没有跨组件事务锁，并发写入中的查询可能观察到更新前快照。

恢复收敛

Store 是最终事实来源。即使进程内派生状态发生偏差，重启恢复会从当前记录重新生成 host shadow 和 GPU index。

语义回退

只有过滤或查询类型不能由 cuVS 安全表示时才回退 native；这是能力路由，不是异常吞噬。

运行错误

缺少 cuVS/CuPy、无可见 GPU、CUDA 初始化失败、build/search 异常均直接报错，不静默切到 CPU，以免线上配置错误被掩盖。

失败矩阵

场景当前行为理由
无 cuVS/CuPy 或无 CUDA device初始化失败启用 cuVS 就应确认 GPU 路径真实生效。
向量维度错误ValueError防止构建不可解释的索引状态。
不支持的过滤 + fallback=truenative保持完整查询语义。
不支持的过滤 + fallback=false显式报错调用方选择了严格 GPU 路径。
GPU build/search 运行异常异常上抛不把资源不足、驱动或数据问题伪装成正常降级。
空数据集 / filter 无候选返回空结果正常边界情况。
已知事务窗口： 当前写链路是 Store → native index → cuVS shadow。若进程在中间异常退出，重启可以从 Store 收敛； 但进程不退出且 cuVS shadow 更新失败时可能暂时分叉。性能版应通过统一 mutation journal、失败后强制 reload， 或构建新快照后原子交换来封闭该窗口。

08. 当前限制与性能画像

限制影响后续方向
每次 mutation 后整体重建写密集负载下构建成本和首查延迟为 O(N)。base + delta、阈值重建、CAGRA extend、双缓冲切换。
host shadow 全量保存向量与字段与 Store/native/GPU 形成多份内存占用。共享连续内存、按需字段列、去除重复 native dense。
每次过滤在 CPU 扫描 N 条记录高 QPS 或低维向量时过滤可能先成为瓶颈。字段列式化、常用租户 bitset cache、GPU/bitmap 索引。
每 index 一把锁同一 index 查询无法充分利用 CUDA stream 并发。不可变 GPU snapshot + 并发 readers，构建单独加锁。
Python/CuPy 数据面存在对象、复制与 GIL 管理开销。基准证明必要后，下沉到 C++ cuVS C API。
恢复需要全量扫描 Store大集合启动/首查成本增加。版本化派生 cache、后台预热、受控序列化。
native dense 仍并存消耗 CPU 内存和写入成本。等 cuVS 覆盖率、稳定性和回退策略明确后再评估裁剪。
CPU int8 / GPU float32显存、精度和 score 不属于等 dtype 对比，少量 top-k 顺序可能不同。保留非侵入式默认；结果报告 dtype、Recall@K 与两侧内存。
GPU shadow 固定 float32大规模 collection 的显存容量受限。先评估 float16；再评估 CAGRA int8/VPQ。native scaled-int8 兼容需要专门的 distance/rerank 路径。

因此当前版本适合“证明 cuVS 能正确接入 OpenViking”，不适合据此给出最终 QPS、P95 或成本结论。 特别是小数据量下，CPU 原生 flat search 可能比 GPU 端到端更快；性能价值必须在真实规模、batch 和过滤分布下测量。

09. 从功能版到性能版

Phase 0 · 功能集成 当前

可配置 adapter、brute-force/CAGRA、bitset prefilter、native fallback、mutation dirty rebuild、恢复重建与 CPU fake-runtime 测试。

Phase 1 · GPU 环境正确性验证

在真实 GPU/CUDA 上运行 smoke；与 native exact 做 top-k/score/filter 对齐；覆盖 upsert、delete、重启恢复和 CAGRA 基础召回。

Phase 2 · 可复现实验基线

固定数据集、维度、规模、query batch、过滤选择性和更新比例，输出 recall@k、QPS、P50/P95/P99、build time、CPU RAM 与 VRAM。

Phase 3 · 数据面性能化

不可变 base + 小型 delta/tombstone、异步阈值重建、双 GPU index 原子切换、bitset cache、批量 search 与资源复用。

Phase 4 · 深度集成（按收益决定）

评估将 cuVS C API 接入 native C++ engine，降低 Python/CuPy 开销；再考虑多 GPU、分片、持久化 cache 和生产可观测性。

建议的性能版目标架构

  • 读路径：查询只读取不可变 GPU snapshot，可并发执行，不被普通写锁阻塞。
  • 写路径：新写进入 host delta；删除进入 tombstone；小规模 delta 用 exact GPU/CPU 补充检索。
  • 合并：达到条数、比例或时间阈值后后台构建新 base，完成后原子切换。
  • 过滤：常用 account_id、path 前缀维护可复用 bitmap，避免逐查询扫描全部字段。
  • 可观测：记录 backend/algorithm、build latency、search latency、fallback reason、eligible count、VRAM、recall sample。

10. GPU 环境验证计划

架构文档评审后再申请卡。第一轮只验证“真实走 GPU 且结果正确”，不在环境和数据未固定前做性能结论。

环境记录

  • GPU 型号、数量、显存、compute capability。
  • driver、CUDA runtime、Python、cuvs-cu12/13、CuPy 版本。
  • OpenViking commit、配置文件和运行命令。

功能验收

  • examples/cuvs_smoke.py 在真实 cuVS runtime 上通过。
  • brute-force 与 float32 ground truth 在 cosine/IP/L2 下对齐；对 native int8 单独报告 Recall@K 和 score 差异。
  • account_id、path、range、contains、and/or/must_not 过滤不越权且结果一致。
  • date_time/geo_point 确认进入 native fallback；关闭 fallback 时返回清晰错误。
  • upsert 后可搜到新值、delete 后不可搜到，连续多次变更后结果正确。
  • 持久化 Collection 关闭再打开，首次查询能重建 GPU index 并得到一致结果。
  • CAGRA 能构建和检索，基础 recall@k 相对 brute-force 达到预设门槛。
  • 无 GPU 或制造 CUDA 错误时不静默走 native。

12. 关键决策与备选方案

方案结论原因
把 cuVS 做成完整独立 VectorDB backend不采用cuVS 不提供完整 CRUD、标量、稀疏、事务和数据库持久化，会复制大量 OpenViking 逻辑。
在 LocalIndex 内替换 pure dense executor当前方案边界最小，能复用 Store、恢复、回表与 native 能力补集。
为 cuVS 改写 native 默认量化拒绝会改变既有 CPU 行为；当前保留 native int8，并让 GPU 独立维护 float32 shadow。
直接在 native C++ engine 链接 cuVS C API后续评估数据面更短，但 CUDA 工具链、wheel/ABI、构建矩阵和维护成本更高，应由 benchmark 证明必要性。
未过滤 top-k 后做 post-filter拒绝无法保证过滤后 top-k 完整性，尤其对高选择性租户过滤不安全。
持久化 cuVS index 作为权威数据拒绝恢复能力会被 GPU、CUDA/cuVS 版本和格式兼容性绑定；可做 cache，但不能做事实来源。
首版默认 CAGRA不采用ANN 会混入召回误差和参数影响；brute-force 更适合功能对齐与 ground truth。
